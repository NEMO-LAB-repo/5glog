# Data Layout

This app is moving toward a data-driven structure so new events and logcodes can be added without rewriting UI logic.

## Events

- `events/index.json` / `events/index.js`: event catalog shown on the Event page.
- `events/handover.json` / `events/handover.js`: handover procedure metadata and step-to-logcode/field mapping.

For a new event, add one entry to `events/index.*` and create a matching event data file:

```json
{
  "id": "ue_registration",
  "title": "UE Registration",
  "steps": {
    "1": {
      "title": "1. Example Step",
      "layer": "RRC",
      "logcodes": ["0xB821"],
      "fields": ["exampleField"]
    }
  }
}
```

## Logcodes

Logcodes are split into one file per logcode:

- `logcodes/0xB821.json`
- `logcodes/0xB952.json`
- `logcodes/...`

Indexes are generated by `../scripts/build-indexes.js`:

- `generated/logcode_index.json`
- `generated/logcode_bundle.json`
- `generated/field_index.json`
- `generated/field_to_logcodes.json`
- `generated/event_to_logcodes.json`
- `fields/<field-id>.json`

The React app uses `logcode_index.json` on initial load, then loads `logcodes/<code>.json` only when a user opens a message detail. The Field page uses the lighter `field_index.json`; `field_to_logcodes.json` is retained for the legacy static page and full-data export.

The matching `.js` files expose browser globals so the app can still run directly from a static server or `file://`.

To rebuild after adding logcodes or events:

```sh
node scripts/build-indexes.js
```

## Overview content (message & field pages)

Every message page and every field page shows a set of boxes. Some boxes fill in
automatically; the rest you author by hand in **two overlay files**. Do **not**
edit `logcodes/*.json` or `generated/*.json` for this — those are regenerated by
`build-indexes.js` and your edits would be lost.

| You want to fill | Edit this file | Key to use |
| --- | --- | --- |
| A message (logcode) page | `message_overviews.json` | the message `id` from `generated/logcode_index.json` (e.g. `0xb822-nr5g-rrc-mib-info`) |
| A field page | `field_overviews.json` | the field `id` from `generated/field_index.json` |

Value precedence for each box: **in-app edit (saved in the browser) → overlay
file → empty** (`"No content yet."`). The overlay file is the shared default
that gets committed and published; an in-app edit only overrides it locally.

### What a logcode (message) needs

All keys are optional — any box you leave out renders empty.

```json
{
  "0xb822-nr5g-rrc-mib-info": {
    "description": "What this message reports and how to read it.",
    "relatedPairs": "0xB96A | how B96A relates to this message\n0xB97F | another related message"
  }
}
```

- `description` — free text shown in the "Message description" box.
- `relatedPairs` — the "Related messages" box. **One entry per line**, format
  `<logcode> | <relation text>`. A line whose logcode matches a known message
  becomes a clickable link.

The "Logcode Structure" and "Related Event" boxes are filled automatically (from
the logcode's `detail` and from event data), so they are not authored here.

### What a field needs

```json
{
  "<field-id>": {
    "description": "What this field means and how to read it.",
    "range": "0..1023, or an enum of allowed values.",
    "related": "0xB822, 0xB823",
    "similar": "OtherFieldName, AnotherField"
  }
}
```

- `description` — free text shown in the "Field description" box.
- `range` — the "Value range" box.
- `related` — the "Related logcode" box, comma- or newline-separated. If omitted,
  it falls back to the logcodes the field already appears in.
- `similar` — the "Similar fields" box, comma- or newline-separated. Names that
  match a known field become clickable links.

### How to update

1. Find the `id` in the matching `generated/*.json` index.
2. Add or edit an entry in `message_overviews.json` / `field_overviews.json`.
3. Save — changes show up on the next `npm run dev` reload. **No rebuild needed**;
   these overlay files are read directly and are not generated.

Keys starting with `_` (the `_comment` / `_example` entries) are ignored by the
app — leave them as inline documentation or delete them.

> Note: if a message/field `id` later changes in the source data, an overlay
> entry keyed by the old `id` silently stops matching (the box just goes empty,
> no error). Key entries by the current id in `generated/*.json`.

## Notes

Team notes live in `notes/message_notes.json` and `notes/field_notes.json`.

- Message note keys are logcode record ids from `generated/logcode_index.json`.
- Field note keys are field ids from `generated/field_index.json`.
- Edit notes in a branch and open a GitHub pull request. The public site is static, so notes entered in the browser are not written back to GitHub.

See `notes/README.md` for examples.
